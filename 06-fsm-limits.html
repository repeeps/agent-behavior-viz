PART 1 · FINITE STATE MACHINE

FSM의 한계 — 전이 폭발

FSM은 단순할 땐 명쾌해요. 하지만 상태가 늘어나면 전이(화살표)가 제곱으로 폭발하고, 전역 이벤트가 모든 상태를 오염시키며, 한 덩어리를 다른 곳에 떼어 쓰기도 어렵죠. 이 한계들이 바로 Behavior Tree(PART 2)로 넘어가는 다리예요. 직접 슬라이더를 움직여 폭발을 눈으로 봅시다.

Step 00 — Transition Explosion

상태가 늘면 전이가 폭발한다

"모든 상태에서 모든 상태로 갈 수 있다"면, 상태 수 N에 대해 전이 수는 N×(N−1)이에요. 각 상태가 자기 자신을 뺀 나머지 N−1개로 가는 화살표를 갖기 때문이죠. 슬라이더로 N을 키워 보세요 — 화살표가 순식간에 빽빽해져요.

4
전이 수 = N × (N − 1) =

→ N=2면 전이 2개지만, N=8이면 개예요. 상태를 2배로 늘리면 전이는 약 4배로 늘죠(제곱 증가). 사람이 손으로 관리할 수 있는 수준을 금방 넘어섭니다.

Step 01 — Global Event Nightmare

전역 이벤트의 악몽

"어디서든 사망하면 → 리스폰" 같은 전역 전이를 떠올려 보세요. FSM엔 "모든 상태"라는 개념이 없어서, 상태 하나하나마다 똑같은 화살표를 일일이 그어줘야 해요. 전역 이벤트 1개를 추가할 때마다 전이가 상태 수에 비례해(최대 N−1개씩) 더 늘어나요. 상태가 많아질수록 이 비용이 누적돼 그래프가 화살표로 뒤덮입니다.

→ 기본 전이(파랑) 위에 전역 이벤트(노랑)가 모든 상태에 하나씩 달라붙어요. 전역 이벤트 G개면 전이가 N×G개 추가됩니다. 하나만 빠뜨려도 "거기선 죽어도 리스폰이 안 되는" 버그가 생기죠.

Step 02 — No Reuse

재사용이 안 된다

FSM의 전이는 특정 상태에 묶여 있어요. "전투 행동" 한 덩어리를 다른 캐릭터·다른 맥락에 떼어 쓰려 해도, 그 전이들이 원래 상태 이름을 가리키고 있어서 그대로 복사하면 깨집니다. 모듈성이 없는 거죠.

묶임 Coupling

전이 = 상태쌍

전이는 항상 (from, to) 상태 쌍에 묶여요. 상태를 옮기면 모든 화살표를 다시 이어야 해요.

복붙 실패

떼면 깨진다

"전투" 부분만 복사해도, 가리키던 idle·flee 같은 외부 상태가 없으면 무효 전이가 돼요.

계층 없음

평평한 그래프

FSM은 한 장의 평면 그래프예요. "이 묶음을 하나의 부품으로" 다룰 단위가 없죠.

한눈에

상태 5개짜리 "전투 FSM"을 새 프로젝트에 붙이려면 → 전이가 가리키는 외부 상태를 다 찾아 다시 연결하고, 전역 이벤트도 새 상태마다 다시 그어야 해요. 떼어 쓰는 게 새로 만드는 것만큼 힘들다면, 그건 모듈이 아니에요.

Step 03 — Enter Behavior Tree

그래서, Behavior Tree

Behavior Tree(BT)는 행동을 트리(나무)로 표현해요. 가지(subtree)를 통째로 떼어 재사용하고, 우선순위·반응성을 구조 자체로 표현하죠. FSM이 막혔던 지점들을 BT가 어떻게 푸는지 미리 봅시다 — 자세한 건 PART 2(07~)에서.

→ 위는 같은 "도망 우선, 아니면 순찰" 로직을 BT로 그린 모습이에요. ?(Selector)가 왼쪽부터 시도해 우선순위를 표현하고, 각 가지는 독립 부품이라 다른 트리에 그대로 끼워 넣을 수 있어요.

FSM의 약점BT의 해법
전이 폭발 (N×(N−1))전이 자체가 없음 — 매 틱 루트부터 평가, 노드 수만큼만 자람
전역 이벤트를 상태마다 복제우선순위 가지를 위쪽에 한 번만 두면 항상 먼저 검사됨
재사용 불가 (상태에 묶임)서브트리를 통째로 떼어 다른 트리에 끼움 (모듈성)
평평한 그래프, 계층 없음트리 = 계층 구조. 복잡함을 가지로 캡슐화
반응성 표현이 어색매 틱 재평가 → 상황 변화에 자동 반응
다리를 건너며

FSM은 "지금 어느 상태인가"를 기억으로 들고 갔어요. BT는 "지금 무엇을 해야 하나"를 매 순간 트리를 타고 내려가며 결정해요. 같은 문제를 다른 각도로 푸는 거죠. PART 2에서 이 트리를 직접 틱(tick)시켜 봅니다.
